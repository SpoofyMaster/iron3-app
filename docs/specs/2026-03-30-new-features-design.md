## Feature 1: XP Decay (Hard Mode)

### Rules
- After **3 days** of inactivity in a discipline → lose **10% of that discipline's XP per week** (prorated daily = ~1.43%/day)
- Minimum floor: **0 XP** — can never go negative
- Decay is calculated at app launch and on each daily background check
- Each discipline is tracked independently (can be decaying swim while gaining bike)
- Show a warning notification/banner when decay is active on a discipline

### Implementation
**New file: `lib/xpDecay.ts`**
```ts
export interface DecayState {
  lastSwimDate: string | null   // ISO date of last swim activity
  lastBikeDate: string | null
  lastRunDate: string | null
}

const DECAY_GRACE_DAYS = 3          // days before decay starts
const DECAY_RATE_PER_WEEK = 0.10   // 10% per week
const DECAY_RATE_PER_DAY = DECAY_RATE_PER_WEEK / 7

export function calculateDecay(
  points: number,
  lastActivityDate: string | null,
  referenceDate?: Date
): { newPoints: number; decayApplied: number; daysInactive: number } 
```

**Store changes (`store/useAppStore.ts`):**
- Add `lastSwimDate`, `lastBikeDate`, `lastRunDate` fields (set on each workout log)
- Add `applyXpDecay()` action — called on app launch and after midnight
- Update `logWorkout()` to update the per-discipline last-activity date
- Update `fetchActivities()` to compute per-discipline last dates from Supabase

**Supabase:** Add `xp_decay_log` table to track decay events for audit trail (optional, phase 2)

**UI warnings:**
- On home screen: show orange warning badge on discipline card if decay is active
- Badge text: "⚠️ Swim: -10% this week (inactive 5d)"
- On discipline screens (swim/bike/run tabs): show decay banner at top

## Feature 3: Dynamic PREP Plan

### Concept
When a user selects a target race (Feature 4), the system generates a full periodized IRONMAN preparation plan from today to race day. The plan is:
- **Structured in weeks** — each week has a theme (Base, Build, Peak, Taper)
- **Dynamically generated** based on weeks remaining and user's discipline levels
- **Editable** — user can swap sessions, mark complete, adjust volume
- **Multi-month navigation** — calendar shows full prep period, can scroll month to month

### Data Types (add to `types/index.ts`)
```ts
export interface PrepPlanWeek {
  weekNumber: number           // 1, 2, 3...
  weekOf: string               // ISO date (Monday of that week)
  phase: 'Base' | 'Build' | 'Peak' | 'Taper' | 'Race Week'
  theme: string                // e.g. "Aerobic Foundation", "Race Simulation"
  totalHours: number           // planned training hours
  sessions: PrepSession[]
}

export interface PrepSession {
  id: string
  day: string                  // 'Mon' | 'Tue' | etc
  discipline: Discipline | 'rest' | 'brick'
  type: 'easy' | 'tempo' | 'intervals' | 'long' | 'race_sim' | 'rest'
  durationMin: number
  distanceTarget?: number
  notes: string
  completed: boolean
  isEditable: boolean
}

export interface PrepPlan {
  id: string
  raceId: string               // linked to selected IronmanEvent
  raceName: string
  raceDate: string
  createdAt: string
  totalWeeks: number
  weeks: PrepPlanWeek[]
  isAutoGenerated: boolean     // false = user built manually
  currentWeekIndex: number     // 0-based
}
```

### Plan Generation (`lib/prepPlanGenerator.ts`)
```ts
export function generatePrepPlan(
  raceDate: string,
  startDate: string,
  raceDistance: 'Full' | '70.3',
  swimLevel: DisciplineLevel,
  bikeLevel: DisciplineLevel,
  runLevel: DisciplineLevel,
  weeklyAvailabilityHours: number
): PrepPlan
```

**Periodization logic:**
- Weeks remaining ≥ 20: 60% Base, 25% Build, 10% Peak, 5% Taper
- Weeks remaining 12-19: 40% Base, 35% Build, 15% Peak, 10% Taper
- Weeks remaining 8-11: 20% Base, 40% Build, 25% Peak, 15% Taper
- Weeks remaining 4-7: 10% Base, 30% Build, 35% Peak, 25% Taper
- Weeks remaining 1-3: 100% Taper/Race Week
- Each week: progressive volume increase (10% rule) with deload every 4th week
- Session distribution based on discipline levels (weaker discipline gets more focus)

### Store changes
```ts
// Add to AppState:
prepPlan: PrepPlan | null
selectedRaceEvent: IronmanEvent | null

// Actions:
setPrepPlan: (plan: PrepPlan | null) => void
selectRaceEvent: (event: IronmanEvent) => void
updatePrepSession: (weekIndex: number, sessionId: string, updates: Partial<PrepSession>) => void
markPrepSessionComplete: (weekIndex: number, sessionId: string) => void
getCurrentPrepWeek: () => PrepPlanWeek | null
```

### UI (existing `app/(tabs)/profile.tsx` or training plan screen)
- **Prep Plan tab/section**: shows current week prominently + full plan scrollable
- **Week navigation**: prev/next week arrows + month view toggle
- **Month calendar view**: full prep period visible, color-coded by phase
  - Base = blue, Build = orange, Peak = red, Taper = green
- **Session card**: tap to mark complete, long-press to edit duration/notes
- **Phase badge**: "Week 12 of 24 — BUILD PHASE"

## Feature 4: Race Calendar — Select Target Event

### Flow
1. User opens Race Calendar (`app/events/index.tsx`)
2. Each event card gets a **"Select as Goal" button** (or tap to expand → select)
3. Tapping "Select as Goal" shows modal:
   ```
   🏆 IRONMAN Barcelona 2026
   📅 October 4, 2026 · 187 days away
   
   Do you want an auto-generated PREP Plan?
   
   [✅ Yes, build my plan]  [❌ No, I'll plan myself]
   ```
4. **If Yes**: calls `generatePrepPlan()` → saves to store → navigates to Prep Plan screen
5. **If No**: creates empty PrepPlan with `isAutoGenerated: false` → user fills manually
6. Selected event is saved to store as `selectedRaceEvent`

### Homepage changes (`app/(tabs)/index.tsx`)
- If `selectedRaceEvent` exists → show **TARGET RACE CARD** below the rank card:
  ```
  🎯 TARGET RACE
  IRONMAN Barcelona 2026
  🇪🇸 Barcelona, Spain
  ████████░░░░░░░  187 days · 26 weeks
  [View Prep Plan →]
  ```
- Race card uses orange gradient, shows countdown + progress bar (% of prep time elapsed)
- Tap → navigates to prep plan

### Event card UI update
- Add "selected" state: green checkmark + "YOUR GOAL RACE" badge when selected
- Only one race can be selected at a time — selecting new one prompts "Replace current goal?"

### Supabase persistence
- Save `selected_race_id`, `prep_plan` (JSON) to user profile table
- Load on `fetchProfile()`
